# Add pagination and sorting to the Owners grid

Implements [gh #25](https://github.com/victorrentea/petclinic/issues/25).

## Why

`GET /api/owners` returns every owner as a plain `List<OwnerDto>`, with each owner's pets and
visits lazily loaded per row. With the 28-row dev seed this costs ~61 queries; production is
planned for **~10,000 owners**, where the same call is ~22,000 queries and a full grid render in
the browser — a production incident waiting for the first real dataset.

Issue #25 asks for a sortable, paginated grid. #25 is therefore also the first feature that makes
row *ordering* user-visible, which is the honest moment to fix the two ordering defects underneath
it: the database uses the `C` (byte-order) collation, and no sorted column is unique.

## What Changes

- **BREAKING** — `GET /api/owners` returns a page envelope `{content, totalElements, totalPages,
  number, size}` instead of a bare array. All four in-repo consumers are updated in this change
  (`owner.service.ts`, generated `api-types.ts`, `openapi.yaml`, `petclinic-ui-test/.../api-client.ts`).
  No external consumers exist; the break is fully contained.
- Server-side paging, sorting and the existing `lastName` prefix filter, driven by Spring
  `Pageable` + springdoc `@ParameterObject`.
- Page size is capped server-side at 20 (`max-page-size`), defaults to 10; an unknown `sort`
  property returns **400**, not a 500 stack trace.
- The server expands the requested sort into a deterministic chain and always appends `id`, so
  pages cannot duplicate or drop rows across ties (`city` has London ×7 today).
- Only **Name** and **City** are sortable — a conscious deviation from the issue's literal "any
  column" (see Impact).
- New Flyway migration: recollate `last_name`, `first_name`, `city` to `ro-RO-x-icu` and add the
  two covering indexes (`owners` has only its primary key today).
- N+1 removed by `hibernate.default_batch_fetch_size=50` — ~46 queries per page becomes ~3, with
  no contract change. Explicitly **not** `JOIN FETCH` + `Pageable`, which silently paginates in
  memory.
- Owners grid gains clickable sort headers on Name/City and a page-size selector `[5, 10, 20]`,
  built with the existing Bootstrap table markup (no `mat-table`). Name renders as
  `lastName firstName`. List state lives in URL query params.

## Capabilities

### New Capabilities
- `owners-list-api`: the paged, sorted, filtered `GET /api/owners` contract — envelope shape,
  parameter validation and guards, deterministic total ordering, and query-count behaviour.
- `owners-grid-ui`: the Owners list screen — sortable column headers, pager and page-size control,
  URL-as-state, search/paging interaction, and the empty state.

### Modified Capabilities
None — `openspec/specs/` is currently empty, so both capabilities above are introduced fresh.

## Impact

**Backend** (`petclinic-backend`)
- `OwnerController` / owner service + repository query methods
- New `PageDto<T>` record in `rest/dto/`
- `@RestControllerAdvice`: map `PropertyReferenceException` → 400
- `application.properties`: `spring.data.web.pageable.max-page-size=20`, `default-page-size=10`,
  `spring.jpa.properties.hibernate.default_batch_fetch_size=50`
- New migration `V9__recollate_and_index_owners.sql`

**Contract**
- `openapi.yaml` regenerated by `OpenApiExtractorTest`; `api-types.ts` regenerated by
  `npm run generate:api`. Both are CI drift-checked (`GUARDRAILS.md`) and must never be hand-edited.

**Frontend** (`petclinic-frontend`)
- `owner.service.ts`, `owner-list.component.{ts,html,css}`, existing unused `owners/owner-page.ts`
  interface and the already-present `.owners-controls` / `.owners-pagination` CSS get adopted.
- Latent bug this change introduces if unhandled: the no-results message tests `*ngIf="!owners"`,
  and an empty `content` array is truthy.

**UI tests** (`petclinic-ui-test`)
- `tests/support/api-client.ts` unwraps `.content`; `owner-search.feature` must stay green.

**Deliberately out of scope**
- Address and Telephone sorting: address is free text with leading house numbers (`4` sorts after
  `30`) and a third have none; telephone is a nullable, format-free `text` blob. A sort that
  produces gibberish reads as broken. To be confirmed on the PR as a deviation from #25.
- The other five list grids (Vets, Pets, Visits, Specialties, PetTypes) share the same latent
  problem and are to be filed separately.
- A slim list projection that drops `visits` from the payload — batch fetching makes it
  unnecessary for now, though it remains the more correct long-term shape.
