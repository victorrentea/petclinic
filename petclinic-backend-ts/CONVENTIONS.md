# PetClinic NestJS Port — Conventions

This module (`petclinic-backend-ts`) is a 1:1 port of the Spring Boot backend
at `../petclinic-backend` to **NestJS 10 + TypeORM 0.3 + PostgreSQL** with
TypeScript strict mode. Mirror the Java design exactly. Read the corresponding
Java source before writing any TypeScript.

## Architecture (mirror Java — NO service layer)

- **Controllers inject TypeORM repositories directly** via
  `@InjectRepository(Entity)` and run queries with the repository API or
  `createQueryBuilder`. There is **no service layer**, matching the Java app.
- **Mappers are STATELESS plain functions/classes** in
  `src/<domain>/<domain>.mapper.ts` — NO Nest DI, NO `@Injectable`. This mirrors
  MapStruct's stateless nature and deliberately avoids circular DI between the
  Owner / Pet / Visit mappers. Controllers import mapper functions directly.
- **One feature module per domain** (`owners`, `pets`, `vets`, `visits`,
  `specialties`, `pet-types`, `users`). Each registers its entities via
  `TypeOrmModule.forFeature([...])`. The root `app.module.ts` (owned by the
  Integration phase) imports them.


## Entities & schema

- TypeORM entities live in `src/<domain>/<domain>.entity.ts` (matched by the
  glob `src/**/*.entity.ts`).
- **Table & column names are IDENTICAL to the Java backend (snake_case):**
  `owners, pets, types, visits, vets, specialties, vet_specialties, users,
  roles`; columns `first_name, last_name, birth_date, visit_date, type_id,
  owner_id, pet_id`, etc. Use explicit `@Column({ name: '...' })` where the TS
  property name differs.
- PKs are `INT GENERATED BY DEFAULT AS IDENTITY`.
- `synchronize: false` — **never** auto-DDL. The schema is owned by the
  migrations under `src/migrations/` (ported 1:1 from the three Flyway SQL
  files). Run with `npm run migration:run`.

## Dates

- Java `LocalDate` → TypeORM `'date'` columns; expose as ISO `'YYYY-MM-DD'`
  strings in DTOs (never `Date` objects with time/zone).
- **Domain-core decision (entities + DTOs):** `LocalDate` fields are typed as
  `string` everywhere — `Pet.birthDate`, `Visit.date` (entities, `@Column({type:'date'})`)
  and `PetDto.birthDate`, `Pet/VisitFieldsDto.birthDate/date`, `VisitDto.date`
  (DTOs). TypeORM's `'date'` column already returns a `'YYYY-MM-DD'` string from
  pg, so no conversion layer is needed in mappers. ISO syntax on nullable DTO
  date fields is enforced with `@Matches(/^\d{4}-\d{2}-\d{2}$/)`.
- `@PastOrPresent` is ported as a custom validator
  `src/common/validation/is-past-or-present.validator.ts` (`@IsPastOrPresent`)
  operating on the `'YYYY-MM-DD'` strings; used by `PetDto` / `PetFieldsDto`
  with the message "Birth date must not be in the future".
- **Default Visit date = today:** Java sets `LocalDate date = LocalDate.now()`
  as a field initializer. TypeORM has no equivalent, so `Visit` exposes
  `Visit.create()` and `ensureDate()` (plus the exported `todayIsoDate()`
  helper) — the visit-creation path must call one of these to reproduce the
  default. The DB column itself stays nullable (matches the migration).
- **Preserve sorting:** `Owner.pets` sorted by `name` asc (case-insensitive,
  via `Owner.getPets()`); `Pet.visits` sorted by `visit_date` desc (via
  `Pet.getVisitsSortedByDate()`). Mappers must use these helpers, not the raw
  relation arrays, to match Java's `PropertyComparator` output.

## Known faithful-port caveat: VetFieldsDto regex

`VetFieldsDto.firstName`'s Java `@Pattern` is `^\p{L}+([ '-][\p{L}]+){0,2` —
the `{0,2` is truncated (no closing brace), so it is **not a valid regex** and
`Pattern.compile` throws. It never surfaces in Java because no controller
validates `VetFieldsDto` (the Vet controller uses `VetDto`). The TS port keeps
the broken patterns verbatim but passes them to `@Matches` as **strings** so
they compile lazily (inside `validate()`), exactly like Hibernate Validator —
importing the class never crashes; only a (non-existent) validation path would.

## DTOs & validation (class-validator mirrors jakarta.validation 1:1)

DTO classes live in `src/<domain>/dto/`. Map annotations one-for-one:

| Java (jakarta)        | TypeScript (class-validator)              |
|-----------------------|-------------------------------------------|
| `@NotEmpty`           | `@IsNotEmpty()`                           |
| `@NotNull`            | `@IsDefined()` / `@IsNotEmpty()`          |
| `@Size(min,max)`      | `@Length(min,max)` / `@MaxLength(max)`    |
| `@Min(n)`             | `@Min(n)`                                 |
| `@Pattern(regexp)`    | `@Matches(/regexp/)`                      |
| `@PastOrPresent`      | custom past-or-present date check         |
| `@Digits(int,frac)`   | `@Matches(/.../)`                         |

- Use `@ApiProperty` from `@nestjs/swagger` mirroring the Java `@Schema`
  (examples / descriptions / `accessMode = READ_ONLY` → `readOnly: true`).
- Use class-transformer `@Type(() => NestedDto)` for nested DTOs.

## Endpoints, status codes & errors — match Java EXACTLY

- Paths, HTTP verbs, and status codes must match the Java controllers exactly.
  **Read the controller first.** Examples:
  - `POST` → `201 Created` with a `Location` header.
  - `DELETE` → `204 No Content` (where the Java controller does so).
  - `PUT` → `200` / void as in Java.
  - not found → `404`; validation failure → `400`.
  Use `@HttpCode(...)` and set the `Location` header explicitly when needed.
- **Error responses are RFC-7807 ProblemDetail JSON:**
  `{ type, title, detail, status, timestamp, errors? }`, matching
  `ExceptionControllerAdvice` + the `ValidationError*Extractor` humanized
  messages. (Integration phase wires the global exception filter.)

## Security (mirror @PreAuthorize)

- Role-based access via a `@Roles()` decorator + `RolesGuard`, mirroring
  `@PreAuthorize` `hasRole(@roles.X)` / `hasAnyRole` / `permitAll`.
- HTTP **Basic** auth backed by the `users` / `roles` tables, passwords bcrypt
  (`bcryptjs`). The seeded `admin` user has bcrypt hash for `admin`.
- **Enabled only when `PETCLINIC_SECURITY_ENABLE=true`** (default `false` =
  permit all). See `src/config/app-config.ts` (`isSecurityEnabled()`).
- CORS allows `http://localhost:4200` with credentials (see `main.ts`).

## Code style

- TypeScript **strict** (`strictNullChecks`). No `any` unless unavoidable.
- Constructor injection. Keep lines reasonably short (~120 cols).
- Swagger UI served at `/swagger-ui`.

## Ownership / parallel-work rules

- **Only create/modify files under your assigned folder.**
- **Do NOT edit `src/app.module.ts` or `src/main.ts`** — the Integration phase
  owns those (placeholders for now).
- **Do NOT run a full `npm run build`** while modules are being written in
  parallel; use `npx tsc --noEmit` scoped to your own files instead.
