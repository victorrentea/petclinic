# PetClinic Backend — Conventions

This module (`petclinic-backend-ts`) is the PetClinic backend, built on
**NestJS 10 + TypeORM 0.3 + PostgreSQL** with TypeScript strict mode.

## Architecture (NO service layer)

- **Controllers inject TypeORM repositories directly** via
  `@InjectRepository(Entity)` and run queries with the repository API or
  `createQueryBuilder`. There is **no service layer**.
- **Mappers are STATELESS plain functions/classes** in
  `src/<domain>/<domain>.mapper.ts` — NO Nest DI, NO `@Injectable`. Keeping them
  stateless deliberately avoids circular DI between the Owner / Pet / Visit
  mappers. Controllers import mapper functions directly.
- **One feature module per domain** (`owners`, `pets`, `vets`, `visits`,
  `specialties`, `pet-types`, `users`). Each registers its entities via
  `TypeOrmModule.forFeature([...])`. The root `app.module.ts` imports them.

## Layout

```
src/
  config/        app-config.ts, typeorm.config.ts
  migrations/    TypeORM migration classes (schema + seed data)
  data-source.ts standalone DataSource for the TypeORM CLI

  <domain>/

```

## Entities & schema

- TypeORM entities live in `src/<domain>/<domain>.entity.ts` (matched by the
  glob `src/**/*.entity.ts`).
- **Table & column names are snake_case:** `owners, pets, types, visits, vets,
  specialties, vet_specialties, users, roles`; columns `first_name, last_name,
  birth_date, visit_date, type_id, owner_id, pet_id`, etc. Use explicit
  `@Column({ name: '...' })` where the TS property name differs.
- PKs are `INT GENERATED BY DEFAULT AS IDENTITY`.
- `synchronize: false` — **never** auto-DDL. The schema is owned by the
  migrations under `src/migrations/`. Run with `npm run migration:run`.

## Dates

- `LocalDate`-style date fields map to TypeORM `'date'` columns; expose them as
  ISO `'YYYY-MM-DD'` strings in DTOs (never `Date` objects with time/zone).
- **Domain-core decision (entities + DTOs):** date-only fields are typed as
  `string` everywhere — `Pet.birthDate`, `Visit.date` (entities,
  `@Column({type:'date'})`) and `PetDto.birthDate`, `Pet/VisitFieldsDto.birthDate/date`,
  `VisitDto.date` (DTOs). TypeORM's `'date'` column already returns a
  `'YYYY-MM-DD'` string from pg, so no conversion layer is needed in mappers. ISO
  syntax on nullable DTO date fields is enforced with `@Matches(/^\d{4}-\d{2}-\d{2}$/)`.
- A "past or present" rule is implemented as a custom validator
  `src/common/validation/is-past-or-present.validator.ts` (`@IsPastOrPresent`)
  operating on the `'YYYY-MM-DD'` strings; used by `PetDto` / `PetFieldsDto` with
  the message "Birth date must not be in the future".
- **Default Visit date = today:** the `Visit` entity exposes `Visit.create()`
  and `ensureDate()` (plus the exported `todayIsoDate()` helper) — the
  visit-creation path must call one of these to default the date to today. The
  DB column itself stays nullable (matches the migration).
- **Preserve sorting:** `Owner.pets` sorted by `name` asc (case-insensitive, via
  `Owner.getPets()`); `Pet.visits` sorted by `visit_date` desc (via
  `Pet.getVisitsSortedByDate()`). Mappers must use these helpers, not the raw
  relation arrays.

## Known caveat: VetFieldsDto regex

`VetFieldsDto.firstName`'s original pattern `^\p{L}+([ '-][\p{L}]+){0,2` has a
truncated `{0,2` (no closing brace), so it is **not a valid regex**. It never
surfaces because no controller validates `VetFieldsDto` (the Vet controller uses
`VetDto`). The broken patterns are kept verbatim but passed to `@Matches` as
**strings** so they compile lazily (inside `validate()`) — importing the class
never crashes; only a (non-existent) validation path would.

## DTOs & validation (class-validator)

DTO classes live in `src/<domain>/dto/`. Validation decorators:

| Rule              | TypeScript (class-validator)              |
|-------------------|-------------------------------------------|
| not empty         | `@IsNotEmpty()`                           |
| required          | `@IsDefined()` / `@IsNotEmpty()`          |
| size (min,max)    | `@Length(min,max)` / `@MaxLength(max)`    |
| min value         | `@Min(n)`                                 |
| pattern           | `@Matches(/regexp/)`                      |
| past or present   | custom past-or-present date check         |
| digits            | `@Matches(/.../)`                         |

- Use `@ApiProperty` from `@nestjs/swagger` for examples / descriptions /
  `readOnly: true` on read-only fields.
- Use class-transformer `@Type(() => NestedDto)` for nested DTOs.

## Endpoints, status codes & errors

- Paths, HTTP verbs, and status codes:
  - `POST` → `201 Created` with a `Location` header.
  - `DELETE` → `204 No Content`.
  - `PUT` → `200` / void as appropriate.
  - not found → `404`; validation failure → `400`.
  Use `@HttpCode(...)` and set the `Location` header explicitly when needed.
- **Error responses are RFC-7807 ProblemDetail JSON:**
  `{ type, title, detail, status, timestamp, errors? }`, produced by the global
  exception filter (`src/common/all-exceptions.filter.ts`) with humanized
  validation messages.

## Security

- Role-based access via a `@Roles()` decorator + `RolesGuard`.
- HTTP **Basic** auth backed by the `users` / `roles` tables, passwords bcrypt
  (`bcryptjs`). The seeded `admin` user has a bcrypt hash for `admin`.
- **Enabled only when `PETCLINIC_SECURITY_ENABLE=true`** (default `false` =
  permit all). See `src/config/app-config.ts` (`isSecurityEnabled()`).
- CORS allows `http://localhost:4200` with credentials (see `main.ts`).

## MCP server

- An MCP server is exposed over SSE at `GET /sse` (clients POST messages to
  `/mcp/messages`). See `src/mcp/`.
- Authenticated with `X-API-Key` (maps an API key → owner id) via
  `ApiKeyMiddleware`, scoped to the MCP routes only.

## Code style

- TypeScript **strict** (`strictNullChecks`). No `any` unless unavoidable.
- Constructor injection. Keep lines reasonably short (~120 cols).
- Swagger UI served at `/swagger-ui`.
