# DB Schema → ER Diagram with Red-Highlighted Changes

**Date:** 2026-06-25
**Status:** Approved (design) — pending spec review
**Author:** Victor + Claude

## Problem

Reviewing the database changes of a large feature is draining when the only
artifacts are raw SQL DDL or a `git diff` of the schema file. We want a
visual: a PlantUML **Entity-Relationship diagram** regenerated automatically
from the canonical schema, with the **changes introduced by the current commit
highlighted in red** so a reviewer sees at a glance what the commit did to the
data model.

## Source of Truth

`DB.sql` (repo root) — the canonical pg_dump-style full schema, already
regenerated by `DbSchemaExtractorTest` (boots Postgres, runs Flyway, dumps the
schema). It is a single self-contained file containing `CREATE TABLE`,
`ALTER TABLE … ADD CONSTRAINT` (PK / UNIQUE / FK), and `CREATE INDEX`. This is
"the file" the diagram is generated from. The diagram is a **consumer** of
`DB.sql`; it does not change how `DB.sql` is produced.

## Fit With Existing Conventions

This repo already runs a "living documentation" pipeline (see `GUARDRAILS.md`):

- `*ExtractorTest`s regenerate artifacts into `petclinic-backend/docs/generated/`
  (e.g. `DomainModel.puml` from JPA annotations).
- **pre-commit** *generates + auto-stages* derived artifacts, gated on their
  input being staged (the `openapi.yaml → api-types.ts` step is the template).
- **pre-push** and **CI** *drift-check* `petclinic-backend/docs/generated`,
  `openapi.yaml`, and `db.sql` against `HEAD`.

This feature mirrors that pattern: the ER diagram is a generated artifact in
`docs/generated/`, produced and staged by pre-commit when `DB.sql` is staged,
and drift-checked thereafter.

## Components

### 1. Parser — sqlglot (Python)

`sqlglot` is the SQL parser. Rationale, verified by probing both candidates
against the real `DB.sql`:

- **node-sql-parser** hard-crashes on PostgreSQL `ALTER TABLE … ADD GENERATED
  BY DEFAULT AS IDENTITY (…)` — aborts the whole parse. Rejected.
- **sqlglot** parses the entire dump: all 10 tables with columns / types /
  NOT-NULL, primary keys, and foreign-key relationships extracted cleanly;
  unsupported statements (`CREATE EXTENSION`, identity `ALTER`s, gin indexes)
  degrade gracefully to opaque `Command` nodes instead of failing. It is the
  professional, gold-standard multi-dialect SQL parser.

Python 3.12 is already present locally. sqlglot is a single pure-Python
dependency, pinned in `requirements.txt`, installed into a self-bootstrapping
venv under the script directory (gitignored). CI does **not** need it (no test
regenerates this file — see Drift & CI below).

### 2. Generator script

`petclinic-backend/docs/scripts/db_schema_to_puml.py`

Inputs:
- `--current <path>`: the new schema SQL (the staged `DB.sql`).
- `--baseline <path-or-empty>`: the previous committed schema SQL, or empty for
  bootstrap.
- `--out <path>`: output `.puml`.

Behavior:
1. Parse both schemas into an internal model: `{table: [columns…], pks, uniques,
   fks: [(table, cols) → (ref_table, ref_cols)]}`.
2. Exclude Flyway's internal `flyway_schema_history` table (not domain data;
   absent from `DomainModel.puml` too).
3. Diff current vs baseline (see Red-Diff Algorithm).
4. Emit a deterministic PlantUML ER diagram (see Output Format).

Determinism: tables sorted alphabetically, columns in dump order, relationships
sorted by `(table, ref_table, cols)`. Same inputs ⇒ byte-identical output.

A thin wrapper `petclinic-backend/docs/scripts/gen-db-schema-diagram.sh`
ensures the venv exists + deps installed, resolves the baseline from git, and
invokes the Python script. The hooks call the wrapper.

### 3. Output format — PlantUML ER

`petclinic-backend/docs/generated/DbSchema.puml`. Uses PlantUML **`entity`**
notation with crow's-foot relationships (information-engineering style) — a
*physical* ER view, complementary to the existing JPA *class* diagram in
`DomainModel.puml`.

- Header: `title Database Schema (ER)` / `caption Generated from DB.sql —
  red = changed in this commit`.
- Each table → an `entity`. Columns rendered as:
  - PK: `* id : integer <<PK>>` (the `*` denotes the primary key).
  - FK: `pet_id : integer <<FK>>`
  - NOT NULL shown as explicit trailing `not null` text (not `*`, to avoid
    colliding with the PK marker).
- Relationships: one crow's-foot line per FK, e.g.
  `owners ||--o{ pets : owns`.

### 4. Red-Diff Algorithm (commit-scoped)

Baseline = the schema at `HEAD` (`git show HEAD:DB.sql`); current = the staged
schema (`git show :DB.sql`). Red marks exactly what *this commit* changes:

- **New table** → entity header + border red.
- **New column** → red column line.
- **Changed column** (type or nullability differs) → red column line.
- **Removed column** → red strikethrough line, retained in the entity.
- **New FK relationship** → red crow's-foot line.
- **Removed FK relationship** → red line labelled `(removed)` (best-effort).

PlantUML coloring uses `#line:red;text:red` on entities and `<color:red>…
</color>` / `<s>…</s>` on column lines; link color via `#red`. Exact tokens
verified to render during implementation.

Bootstrap: if `HEAD:DB.sql` does not exist (first introduction of `DB.sql`),
render with **no** red — a clean baseline snapshot, not an all-red wall.

Hard parse failure (sqlglot raises, not a graceful fallback) ⇒ script exits
non-zero so the commit is blocked rather than committing a broken diagram.

## Hook Wiring

### pre-commit — generate + stage (gated)

Mirrors the existing `openapi.yaml → api-types.ts` step. When `DB.sql` is among
the staged files:

1. Run the wrapper (baseline `git show HEAD:DB.sql`, current `git show :DB.sql`).
2. `git add petclinic-backend/docs/generated/DbSchema.puml`.

So the diagram is committed in the same commit as the schema change and
therefore "pushed along with" it. If `DB.sql` is not staged, the step is
skipped and the diagram is untouched (its red highlights persist until the next
schema-changing commit — no churn).

### pre-push — consistency guard

The existing drift check already covers `docs/generated` vs `HEAD`. Add one
lightweight guard: if the commits being pushed touched `DB.sql` but **not**
`docs/generated/DbSchema.puml`, block with an actionable message (catches a
`git commit --no-verify` that bypassed generation). This guard does **not**
regenerate the diagram (which would recompute red against the now-committed
schema and wipe it).

## Drift & CI Interaction

The committed `DbSchema.puml` is a **source artifact whose red depends on git
history**, so — unlike `DomainModel.puml` — it must **not** be regenerated by an
always-on guardrail test. Concretely:

- Do **not** add a `*ExtractorTest` that rewrites `DbSchema.puml` during
  `mvn test`. Re-running it post-commit would diff the schema against itself
  (HEAD already contains the change) → empty red → drift failure.
- pre-push/CI only *drift-check* the file (working tree vs `HEAD`); since
  nothing regenerates it after pre-commit, it stays consistent and CI needs no
  Python.

## Testing

- **Snapshot:** generate from the current `DB.sql` with baseline == current ⇒
  output contains all 10 expected entities (minus `flyway_schema_history`), the
  6 expected FK relationships, and **no** red markers.
- **Diff cases:** synthetic baseline vs current that (a) adds a column,
  (b) adds a table, (c) adds an FK, (d) changes a column type, (e) drops a
  column ⇒ red markers appear on exactly those elements and nowhere else.
- **Determinism:** two runs on identical inputs ⇒ byte-identical output.
- **Hook behavior:** staging a `DB.sql` change runs the step and stages
  `DbSchema.puml`; staging an unrelated file does not touch it.
- **Render validity (optional):** if `plantuml` is available, the generated
  `.puml` renders without syntax errors.

## Out of Scope (YAGNI)

- Rendering to PNG/SVG (repo keeps `.puml` source only).
- Push-scoped highlighting / PR-comment delta artifacts (rejected option B).
- Indexes, sequences, identity details, check constraints in the diagram
  (ER focuses on tables, columns, keys, relationships).
- Showing removed *tables* as ghost entities (removed columns/relationships are
  handled; whole-table removal is rare and can be a follow-up).
